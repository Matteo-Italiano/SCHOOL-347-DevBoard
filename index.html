Phase 1: Anforderungen & Architektur

A1F Anforderungen erfassen

SMART-LernzielIch kann bis Ende des Projekts die funktionalen und nicht-funktionalen Anforderungen für eine containerisierte Webanwendung (DevBoard) in einer Anforderungsliste dokumentieren.

Bevor wir irgendwas mit Docker machen, müssen wir zuerst wissen was unsere App überhaupt können soll. Ohne klare Anforderungen wissen wir nicht welche Container wir brauchen und wie die miteinander reden.

Wir haben uns für ein einfaches Task-Board entschieden => DevBoard. Damit kann man Tasks erstellen, anzeigen und löschen.

Funktionale Anforderungen

- Tasks anzeigen: eine Übersicht aller Tasks im Board
- Tasks erstellen: neue Aufgaben mit Titel anlegen
- Tasks löschen: erledigte Aufgaben entfernen
- Persistente Datenhaltung: Tasks bleiben nach Neustart erhalten
- REST API: Frontend redet über HTTP mit dem Backend

Nicht-funktionale Anforderungen

- Containerisiert: Jeder Dienst läuft in seinem eigenen Container
- Reproduzierbar: docker compose up startet die gesamte App
- Isolierte Netzwerke: Die Datenbank ist nicht direkt von aussen erreichbar
- Daten-Persistenz: Die DB-Daten liegen auf einem Docker Volume
- Portabilität: Die Anwendung läuft auf jedem System mit Docker

Technologie-Entscheid

Komponente Technologie Warum?
Frontend Nginx + HTML/JS Leichtgewichtig und schnell
Backend Python Flask Einfache REST API, wenig Boilerplate
Datenbank PostgreSQL Robust, Open Source, weit verbreitet
Orchestrierung Docker Compose Perfekt für Multi-Container auf einem Host

B1E Kompositions-Varianten

SMART-LernzielIch kann mindestens 2 Container-Kompositions-Varianten vorschlagen, vergleichen und die gewählte Variante begründen.

Wir haben 3 Möglichkeiten verglichen:

Kriterium Monolith Docker Compose Kubernetes
Was ist das? Alles in einem Container 3 separate Container, ein YAML Jeder Dienst als Pod
Skalierbarkeit Geht nicht Pro Container, ein Host Volle Skalierung
Komplexität Super einfach Geht klar Ziemlich komplex
Wartbarkeit Mühsam Gut Sehr gut

Unser Entscheid: Docker Compose. Beste Balance zwischen Isolation und Einfachheit. Monolith widerspricht dem Container-Prinzip, Kubernetes wäre totaler Overkill.

Backend: Flask API

Das Backend ist eine Python Flask REST API. Die hat Endpoints um Tasks zu erstellen (POST), anzuzeigen (GET) und zu löschen (DELETE). Ausserdem gibt's einen Health-Check Endpoint den Docker benutzt um zu prüfen ob der Container noch lebt.

backend/app.py anzeigen 
from flask import Flask, request, jsonify
from flask_cors import CORS
import psycopg2, os, time

app = Flask(__name__)
CORS(app)

def get_db():
    retries = 5
    while retries > 0:
        try:
            conn = psycopg2.connect(
                host=os.environ.get("DB_HOST", "db"),
                port=os.environ.get("DB_PORT", "5432"),
                database=os.environ.get("DB_NAME", "devboard"),
                user=os.environ.get("DB_USER", "devboard"),
                password=os.environ.get("DB_PASS", "secret"))
            return conn
        except psycopg2.OperationalError:
            retries -= 1
            time.sleep(2)
    raise Exception("Datenbank nicht erreichbar")

def init_db():
    try:
        conn = get_db()
        cur = conn.cursor()
        cur.execute("""CREATE TABLE IF NOT EXISTS tasks (
            id SERIAL PRIMARY KEY,
            title VARCHAR(255) NOT NULL,
            created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP)""")
        conn.commit(); cur.close(); conn.close()
    except Exception: pass

init_db()

@app.route("/api/tasks", methods=["GET"])
def get_tasks():
    conn = get_db(); cur = conn.cursor()
    cur.execute("SELECT id, title, created_at FROM tasks ORDER BY created_at DESC")
    tasks = [{"id": r[0], "title": r[1], "created_at": str(r[2])} for r in cur.fetchall()]
    cur.close(); conn.close()
    return jsonify(tasks)

@app.route("/api/tasks", methods=["POST"])
def create_task():
    data = request.get_json()
    if not data or not data.get("title"): return jsonify({"error": "Titel fehlt"}), 400
    conn = get_db(); cur = conn.cursor()
    cur.execute("INSERT INTO tasks (title) VALUES (%s) RETURNING id, title, created_at", (data["title"],))
    task = cur.fetchone(); conn.commit(); cur.close(); conn.close()
    return jsonify({"id": task[0], "title": task[1], "created_at": str(task[2])}), 201

@app.route("/api/tasks/<int:task_id>", methods=["DELETE"])
def delete_task(task_id):
    conn = get_db(); cur = conn.cursor()
    cur.execute("DELETE FROM tasks WHERE id = %s", (task_id,))
    conn.commit(); cur.close(); conn.close()
    return jsonify({"deleted": task_id})

@app.route("/api/health")
def health():
    try:
        conn = get_db(); cur = conn.cursor()
        cur.execute("SELECT 1"); cur.close(); conn.close()
        return jsonify({"status": "healthy"})
    except Exception as e:
        return jsonify({"status": "unhealthy", "error": str(e)}), 500

Backend: Dockerfile

Basiert auf python:3.12-slim, läuft als nicht-Root-User (appuser) und benutzt Gunicorn als Production-Server.

backend/Dockerfile anzeigen 
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY app.py .
RUN useradd -r appuser
USER appuser
EXPOSE 5000
CMD ["gunicorn", "--bind", "0.0.0.0:5000", "--workers", "2", "app:app"]

Frontend: Nginx als Reverse Proxy

Nginx liefert die statischen Files aus und leitet /api/ Requests an Flask weiter.

frontend/nginx.conf anzeigen 
server {
    listen 80;
    server_name localhost;
    location / {
        root /usr/share/nginx/html;
        index index.html;
    }
    location /api/ {
        proxy_pass http://backend:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

Das Herzstück => die docker-compose.yml bringt alles zusammen. 3 Services, 2 Netzwerke, 1 Volume, Health-Checks, Resource Limits und Restart Policies.

docker-compose.yml anzeigen 
services:
  frontend:
    build: ./frontend
    container_name: devboard-frontend
    ports: ["8080:80"]
    networks: [frontend-network]
    depends_on:
      backend: { condition: service_healthy }
    restart: unless-stopped
    mem_limit: 128m
    cpus: 0.25

  backend:
    build: ./backend
    container_name: devboard-backend
    environment:
      - DB_HOST=db
      - DB_PORT=5432
      - DB_NAME=${DB_NAME}
      - DB_USER=${DB_USER}
      - DB_PASS=${DB_PASS}
    networks: [frontend-network, backend-network]
    depends_on:
      db: { condition: service_healthy }
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://localhost:5000/api/health')"]
      interval: 10s
      timeout: 5s
      retries: 3
      start_period: 10s
    restart: unless-stopped
    mem_limit: 256m
    cpus: 0.5

  db:
    image: postgres:16-alpine
    container_name: devboard-db
    environment:
      - POSTGRES_DB=${DB_NAME}
      - POSTGRES_USER=${DB_USER}
      - POSTGRES_PASSWORD=${DB_PASS}
    volumes: [pgdata:/var/lib/postgresql/data]
    networks: [backend-network]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER} -d ${DB_NAME}"]
      interval: 5s
      timeout: 3s
      retries: 5
    restart: unless-stopped
    mem_limit: 256m
    cpus: 0.5

networks:
  frontend-network: { driver: bridge }
  backend-network: { driver: bridge, internal: true }

Fehlersuche

Um zu zeigen dass wir Fehler selbständig fixen können, haben wir absichtlich 2 Fehler eingebaut:

Fehler 1: Falscher Port im Nginx Proxy
Port in nginx.conf von 5000 auf 3000 geändert => Nginx kann API-Requests nicht weiterleiten => 502 Bad Gateway:

502 Bad Gateway => falscher Proxy-Port

Fix: Port zurück auf 5000, Container neu gebaut => läuft wieder.

Fehler 2: Falsches DB-Passwort
Passwort in .env geändert, aber PostgreSQL hatte das alte noch im Volume => Mismatch:

Backend crashed => dependency failed to start
500 Internal Server Errors

Fix: Passwort zurücksetzen, Volume löschen (docker compose down -v), neu starten.

Reflexion

Was haben wir gelernt?

Wir haben eine komplette containerisierte Webapp von Null aufgebaut. Dabei haben wir nicht nur Docker und Compose besser verstanden, sondern auch gelernt wie wichtig saubere Architektur mit getrennten Netzwerken, Health-Checks und Resource Limits ist.

Am meisten gelernt haben wir bei der Fehlersuche => wenn man absichtlich was kaputt macht und dann mit Docker-Logs rausfinden muss was schief läuft, versteht man plötzlich wie alles zusammenhängt.

Der Cloud-Vergleich war auch spannend. Jetzt verstehen wir warum K8s für grosse Systeme Sinn macht, aber Compose für Entwicklung die bessere Wahl ist.

Trivy war ein Augenöffner: sogar ein frisches Image hat schon Schwachstellen. Regelmässig scannen ist Pflicht.

Was würden wir anders machen?

Mit mehr Zeit hätten wir die App auf Google Cloud Run deployed und eine CI/CD-Pipeline mit GitHub Actions gebaut => Images automatisch bauen, scannen und pushen bei jedem Push auf main. HTTPS mit Let's Encrypt wäre auch nice gewesen.
